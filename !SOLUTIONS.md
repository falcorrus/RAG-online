## [2026-07-05] Внедрение Generative UI v1.1: потоковый рендеринг (SSE) и динамический каталог компонентов

### Описание задачи
Текущий прототип Generative UI в easyFAQ.online страдал от двух ограничений:
1. **Отсутствие стриминга**: Ответ от Gemini отдавался клиенту только целиком. Пользователь долго ждал генерации текста.
2. **Жесткий каталог компонентов**: Набор кастомных HTML-тегов (`<ui-card>`, `<ui-button>` и др.) и правила их использования были жестко зашиты в коде фронтенда [script.js](file:///Users/eugene/MyProjects/easyFAQ.online/script.js) и системном промпте бэкенда.

### Решение
1. **Стриминг ответов (SSE)**:
   - В [server.py](file:///Users/eugene/MyProjects/easyFAQ.online/server.py) реализована новая асинхронная функция `call_gemini_api_stream`, использующая `httpx.stream` для потокового получения токенов от OpenAI-совместимого эндпоинта Google AI Studio или OpenRouter.
   - Эндпоинт `/api/chat` теперь возвращает `StreamingResponse` (тип `text/event-stream`). Полный сгенерированный ответ параллельно накапливается на сервере и сохраняется в логи арендатора (`logs.json`) после закрытия потока.
   - Во фронтенде [script.js](file:///Users/eugene/MyProjects/easyFAQ.online/script.js) функция `processSearch` переписана на чтение SSE-потока через `ReadableStream` и `TextDecoder`.

2. **Динамический каталог компонентов**:
   - Создан централизованный файл конфигурации компонентов [components_catalog.json](file:///Users/eugene/MyProjects/easyFAQ.online/components_catalog.json).
   - На бэкенде добавлена функция `load_genui_rules_from_catalog`, которая при каждом запросе читает каталог и собирает динамический блок правил `GENERATIVE UI RULES` для ИИ, подставляя его в системный промпт [MAIN_PROMPT.md](file:///Users/eugene/MyProjects/easyFAQ.online/MAIN_PROMPT.md) через плейсхолдер `{genui_rules}`.
   - На фронтенде регулярные выражения заменены на инкрементальный парсер `parseGenUI` и универсальный рендерер `renderNode`, который разбирает XML-подобные теги (даже недописанные во время стриминга) и преобразует их в DOM-узлы.

3. **Новый компонент `ui-accordion`**:
   - В каталог компонентов добавлен складной блок `<ui-accordion title="...">содержимое</ui-accordion>`.
   - На фронтенде реализован интерактивный рендеринг аккордеона с переключением видимости контента на Vanilla JS.
   - В [genui.css](file:///Users/eugene/MyProjects/easyFAQ.online/genui.css) добавлены аппаратные стили анимации для раскрытия аккордеона в премиальном стиле.

### Результат
Все изменения зафиксированы в git, а версия проекта повышена до **v1.1**. Пользовательский опыт чата переведен на мгновенный рендеринг.

## [2026-06-26] Интеграция гибридного роутинга запросов (ai.dev с fallback на OpenRouter)

### Описание проблемы
Для экономии квот необходимо делать запросы к ИИ по умолчанию через Stitch API ключ (`ai.dev`). Однако при достижении дневного лимита (ошибка `429 Too Many Requests`) система должна автоматически запоминать состояние исчерпания лимитов до конца дня и переключать все последующие запросы на стабильный платный ключ `OpenRouter` (`easyfaq`), чтобы избежать прерывания обслуживания пользователей.

### Решение
1. **Реализация гибридного роутинга**:
   - В [server.py](file:///Users/eugene/MyProjects/easyFAQ.online/server.py) реализована логика отслеживания лимитов `ai.dev` с помощью текстового маркера в `storage/aidev_limit.txt`, где сохраняется текущая дата в формате `YYYY-MM-DD`.
   - При выполнении ИИ-запроса функция `call_gemini_api` проверяет, есть ли маркер исчерпания лимитов для сегодняшнего дня.
   - Если маркера нет, запрос отправляется на `ai.dev` (Google AI Studio эндпоинт). Если возвращается ошибка `429`, файл `storage/aidev_limit.txt` обновляется текущей датой, и запрос перенаправляется на `OpenRouter`.
   - Если маркер на сегодня уже установлен, запросы сразу идут на `OpenRouter`.
2. **Конфигурация ключей**:
   - В `.env` на локальной машине и на VPS добавлены обе переменные окружения: `AIDEV_API_KEY` (Stitch API key) и `OPENROUTER_API_KEY` (новый ключ OpenRouter с именем `easyfaq`).
   - Использован новый OpenRouter ключ: `sk-or-v1-05271e52...[ЗАМАСКИРОВАНО]`.
3. **Модели**:
   - На обоих направлениях (`ai.dev` и `OpenRouter`) в качестве основной используется модель `google/gemini-2.5-flash` (`gemini-2.5-flash` для Google AI Studio эндпоинта) как наиболее быстрая и стабильная для Generative UI.
4. **Результат**:
   - Продакшн успешно обновлен через `deploy.sh`. Служба `easyfaq-online.service` перезапущена и работает. Запросы по умолчанию идут на бесплатный `ai.dev`, а при исчерпании лимитов прозрачно переключаются на `OpenRouter`.

## [2026-06-26] Ошибка 502 при регистрации нового пользователя и недоступность поддоменов .easyfaq.online

### Описание проблемы
При регистрации нового пользователя через API (например, для поддомена `spain`) возникала ошибка 502 Bad Gateway при попытке открыть ссылку вида `https://spain.easyfaq.online/?token=...`.
Причиной было то, что скрипт настройки SSL `setup_ssl.sh` на VPS конфигурировал Nginx только для поддомена `${subdomain}.rag.reloto.ru`, но не создавал конфигурационный файл и SSL-сертификат для `${subdomain}.easyfaq.online`, который выдается пользователям в качестве основного адреса.

### Решение
1. **Ручное исправление на VPS**:
   - Создан конфигурационный файл Nginx `/etc/nginx/sites-available/spain.easyfaq.online` с проксированием `/api/` на порт 8006 и отдачей статики из `/opt/easyFAQ.online`.
   - Создан симлинк в `sites-enabled`.
   - Получен SSL-сертификат Let's Encrypt для `spain.easyfaq.online`.
   - Файл `/etc/nginx/sites-available/spain.rag.reloto.ru` переписан на чистый 301-редирект на `https://spain.easyfaq.online$request_uri` с использованием SSL.
   - Nginx успешно перезапущен.

2. **Автоматизация**:
   - Переписан скрипт [setup_ssl.sh](file:///Users/eugene/MyProjects/easyFAQ.online/setup_ssl.sh). Теперь он автоматически:
     - Создает временные HTTP-конфиги для обоих доменов (`${subdomain}.easyfaq.online` и `${subdomain}.rag.reloto.ru`).
     - Получает SSL-сертификаты для обоих доменов по отдельности.
     - Переписывает конфиг `${subdomain}.rag.reloto.ru` на постоянный 301-редирект на `https://${subdomain}.easyfaq.online$request_uri`.
     - Применяет настройки в Nginx.
   - Скрипт задеплоен на VPS через `deploy.sh`.

## [2026-06-26] Ошибка "AI connection error" в чате на easyFAQ

### Описание проблемы
Пользователи получали ошибку `AI connection error` при отправке запросов в чат. В логах службы uvicorn на VPS фиксировались ошибки 503 Service Unavailable: `This model is currently experiencing high demand. Spikes in demand are usually temporary. Please try again later.` от эндпоинта Google AI Studio при обращении к модели `gemini-2.5-flash-lite`.

### Решение
1. **Переход на стабильную модель**:
   - В качестве основной модели выбрана `gemini-2.5-flash` вместо `gemini-2.5-flash-lite`, так как она имеет более высокие лимиты и стабильно работает в периоды пиковой нагрузки.
2. **Отказоустойчивость (Fallback)**:
   - В [server.py](file:///Users/eugene/MyProjects/easyFAQ.online/server.py) добавлена общая асинхронная функция `call_gemini_api`. Она сначала пытается выполнить запрос к основной модели `gemini-2.5-flash`. Если запрос завершается ошибкой (код не 200 или исключение сети), она автоматически делает повторную попытку к резервной модели `gemini-2.5-flash-lite`.
   - Все вызовы API (языковая детекция, генерация suggestions и чат-ассистент) переведены на использование этой функции.
3. **Деплой**:
   - Изменения задеплоены на VPS, служба `easyfaq-online.service` перезапущена. Запросы к чату теперь проходят успешно.

   - В [style.css](file:///Users/eugene/MyProjects/easyFAQ.online/style.css) добавлена сетка `.split-layout` (60/40 на десктопе, 1 колонка на мобилках).
   - В [script.js](file:///Users/eugene/MyProjects/easyFAQ.online/script.js) добавлена локализация промо-текстов (RU/EN/PT), логика активации split-layout при `is_main_site == true` и привязка клика по кнопке CTA к открытию онбординга.
2. **Шаблоны документов (Onboarding):**
